# 1. Introduction

The Faym User Payout Management System uses a relational database to manage users, sales, payouts, withdrawals, and payment transactions.

The database has been designed following normalization principles to minimize redundancy while maintaining clear relationships between business entities.

SQLAlchemy ORM is used to map Python classes to relational database tables, allowing the application to remain database-independent and making migration to PostgreSQL or MySQL straightforward.

---

# 2. Database Overview

The database consists of the following entities:

- Brand
- User
- Sale
- Payout
- Withdrawal
- PaymentTransaction

These entities collectively model the lifecycle of a user's earnings and payouts.

# 4. Table Descriptions

---

## 4.1 Brand

Represents a business or organization associated with users.

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| name | String | Brand Name |

### Relationships

One Brand can have multiple Users.

```
Brand

↓

Users
```

Relationship Type:

```
One-to-Many
```

---

## 4.2 User

Represents an individual eligible to receive payouts.

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| brand_id | Integer | Foreign Key → Brand |
| name | String | User Name |
| email | String | Email Address |

### Relationships

A User belongs to one Brand.

A User can have multiple:

- Sales
- Withdrawals

```
Brand

↓

User

↓

Sales

↓

Withdrawals
```

Relationship Type

```
Many-to-One (Brand)

One-to-Many (Sales)

One-to-Many (Withdrawals)
```

---

## 4.3 Sale

Represents a completed sale generated by a user.

This is the primary earning entity.

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| user_id | Integer | Foreign Key |
| amount | Decimal | Sale Amount |
| status | Enum | Pending / Approved / Rejected |
| created_at | Timestamp | Creation Time |

### Relationships

One Sale belongs to one User.

One Sale can generate multiple payouts.

Example

```
Sale

↓

Advance Payout

↓

Final Payout
```

Relationship Type

```
Many-to-One (User)

One-to-Many (Payout)
```

---

## 4.4 Payout

Represents money released against a sale.

Types of payouts include:

- Advance
- Final
- Adjustment

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| sale_id | Integer | Foreign Key |
| amount | Decimal | Payout Amount |
| type | Enum | Advance / Final / Adjustment |
| status | Enum | Pending / Success / Failed |
| created_at | Timestamp | Creation Time |

### Relationships

One payout belongs to exactly one Sale.

A Sale can have multiple payouts.

Example

```
Sale

↓

Advance

↓

Final
```

or

```
Sale

↓

Advance

↓

Adjustment
```

---

## 4.5 Withdrawal

Represents money withdrawn by a user.

Withdrawals reduce the user's available balance.

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| user_id | Integer | Foreign Key |
| amount | Decimal | Withdrawal Amount |
| status | Enum | Pending / Success / Failed |
| created_at | Timestamp | Timestamp |
| completed_at | Timestamp | Completion Time |

### Relationships

A User can create multiple withdrawals.

Each withdrawal generates one payment transaction.

Relationship Type

```
Many-to-One (User)

One-to-One (Payment Transaction)
```

---

## 4.6 PaymentTransaction

Represents the actual payment processing record.

This table exists independently from withdrawals to support future integration with payment gateways.

### Attributes

| Column | Type | Description |
|---------|------|-------------|
| id | Integer | Primary Key |
| withdrawal_id | Integer | Foreign Key |
| transaction_id | UUID | External Transaction Reference |
| status | Enum | Success / Failed |
| remarks | String | Processing Message |
| created_at | Timestamp | Creation Time |

### Relationships

Every payment transaction belongs to exactly one withdrawal.

Relationship Type

```
One-to-One
```

# 7. Database Design Decisions

## Normalization

The schema follows normalization principles by separating independent entities into individual tables.

Benefits:

- Reduced redundancy
- Improved maintainability
- Better scalability
- Easier querying

---

## Foreign Keys

Foreign key constraints ensure referential integrity.

Example:

```
Sale

↓

User

↓

Brand
```

This prevents orphan records.

---

## Enumerations

Status fields use Enumerations rather than strings.

Advantages:

- Prevent invalid values
- Better readability
- Improved consistency
- Easier validation

Examples:

SaleStatus

- Pending
- Approved
- Rejected

WithdrawalStatus

- Pending
- Processing
- Success
- Failed

---

## Decimal for Monetary Values

All monetary values use Decimal instead of floating-point numbers.

Reason:

Floating-point arithmetic can introduce precision errors.

Example

```
0.1 + 0.2

≠

0.3
```

Decimal avoids such inaccuracies and is the preferred choice for financial applications.

# 8. Scalability Considerations

The current schema can be extended without structural changes to support:

- Multiple brands
- Multiple payment gateways
- Audit logging
- Refund processing
- Scheduled payouts
- PostgreSQL migration
- Multi-currency support
- Tax calculations

---

# 9. Summary

The database schema is designed to accurately model the payout lifecycle while maintaining data integrity through normalized tables and well-defined relationships.

The separation of sales, payouts, withdrawals, and payment transactions allows each business entity to evolve independently while preserving a clean and scalable relational structure.

This schema is suitable for production-scale backend applications and can be extended with additional business features without requiring major architectural changes.
